# 贷款行业CRM系统可行性分析报告

> **基于系统**：融鑫汇-客户管理系统（客帮手）深度分析  
> **目标规模**：50万客户数据 / 20名员工并发使用  
> **报告时间**：2026-04-29  
> **报告目的**：评估开发一套高度类似CRM系统的可行性，制定分阶段实施计划

---

## 一、项目概述

### 1.1 系统定位
开发一套面向**金融贷款行业**的顾问型CRM系统，具备以下核心特征：
- 客户全生命周期管理（从分配→跟进→上门→进件→批款→收款）
- 多层级团队管理（公司→大区→部门→顾问）
- 业绩统计与绩效追踪
- 公共池数据分配机制
- 通话与备注系统集成

### 1.2 目标规模
| 指标 | 目标值 |
|------|--------|
| 客户数据总量 | 50万条（含历史数据） |
| 并发用户数 | 20人（峰值30人） |
| 顾问每人持有上限 | 400条客户 |
| 公共池存量 | 约45万条 |
| 每日数据新增 | 约500-1000条 |
| 系统可用性要求 | 99.9%（工作日） |

### 1.3 可行性结论
**结论：可行。** 推荐采用**分阶段实施**策略，预计总工期 **6-9个月**，总成本约 **30-60万元**（含开发+服务器+第三方服务），可根据资金分批投入。

## 三、技术方案选型

### 3.1 推荐技术栈

**理由：** 选择国内成熟技术栈，降低运维难度，利于本地化部署。

#### 前端
| 技术 | 说明 |
|------|------|
| Vue 3 + Vite | 主框架，性能好，生态成熟 |
| Element Plus | UI组件库，表格/表单/弹窗适配良好 |
| Pinia | 状态管理 |
| ECharts | 数据可视化（图表/漏斗图） |
| Axios | HTTP客户端 |

**架构：** 单页应用（SPA），左侧导航+右侧内容区，类似参考系统。内容区使用路由切换（不用iframe，维护更简单）。

#### 后端
| 技术 | 说明 |
|------|------|
| Java Spring Boot | 主框架，稳定可靠 |
| MyBatis-Plus | ORM框架，SQL优化方便 |
| Spring Security + JWT | 认证授权 |
| Redis | 缓存（Session、热点数据、防重复提交） |
| RabbitMQ/Kafka | 消息队列（数据分配、通知） |
| 定时任务（XXL-Job） | 公共池抓取、备忘提醒、统计汇总 |

**备注：** 也可用 Node.js + Express 或 Python + FastAPI，开发速度更快，但Java生态对企业级功能支持更完整。

#### 数据库
| 技术 | 说明 |
|------|------|
| MySQL 8.0 | 主数据库（支持500万行级别） |
| Redis 7 | 缓存、Session、排行榜 |
| MinIO / 阿里云OSS | 文件存储（附件、头像、录音） |

### 3.3 关键设计决策

#### 3.3.1 客户数据分页与索引策略
50万客户数据需要精心设计索引：
- 主键：`customer_id` （BIGINT自增）
- 常用查询索引：`advisor_id`、`status`、`star_level`、`phone_hash`（手机号哈希）、`created_at`、`last_remark_time`
- 分库分表：**初期不需要**，MySQL 5000万行以内性能足够；如后续扩展到1000万+再考虑
- **号码查重**：对手机号建立唯一索引（用哈希存储保护隐私）

#### 3.3.2 公共池高效分配
- 公共池数据单独存储在 `public_pool` 表
- 主顾问客户表 `customer_assignments` 存储分配关系
- 分配操作使用数据库事务 + Redis 分布式锁，防止并发抢同一客户

#### 3.3.3 备注系统设计
```sql
CREATE TABLE customer_remarks (
  id BIGINT AUTO_INCREMENT,
  customer_id BIGINT NOT NULL,
  advisor_id INT NOT NULL,
  content TEXT NOT NULL,
  remark_type TINYINT DEFAULT 0,  -- 0:顾问备注 1:主管点评 2:备忘记录
  created_at DATETIME NOT NULL,
  PRIMARY KEY (id),
  INDEX idx_customer_id (customer_id),
  INDEX idx_advisor_id (advisor_id)
);
```
**规则：** 只允许INSERT，禁止UPDATE/DELETE（不可修改历史）。

### 5.3 拨号APP集成方案

参考系统使用"点拨通"APP，集成方式如下：
```
顾问拨打电话（点拨通APP）
      ↓ 拨打完成后推送
点拨通 Webhook → 我们的API接口 → 写入通话记录表
                                → 更新顾问通话统计
                                → 可触发"未备注提醒"
```

**通话记录数据字段**：
- 顾问ID、客户手机号、通话时长、开始时间、结束时间
- 通话类型（去电/来电）、录音URL

### 5.4 数据安全与合规

| 要求 | 措施 |
|------|------|
| 手机号脱敏展示 | 列表显示 `157****2750`，详情页才显示全号 |
| 数据权限隔离 | 顾问只能看自己名下客户，主管看团队 |
| 操作审计日志 | 记录所有关键操作（谁/何时/做了什么） |
| HTTPS加密传输 | 全站HTTPS，防止数据泄露 |
| SQL注入防护 | 使用参数化查询，禁止拼接SQL |
| 登录安全 | 验证码 + 密码加密存储（bcrypt）|

### 9.3 数据迁移策略

## 九、推荐实施方案

### 9.1 我的推荐方式

基于对现有系统的深度分析，推荐以下方式：

**技术选型**：
- 前端：Vue 3 + Element Plus（比 layui 更现代化、可维护性更强）
- 后端：Spring Boot + MySQL + Redis
- 部署：阿里云/腾讯云 ECS，初期2台小型实例即可

**开发策略**：
1. **先复刻核心**：第一个月完成登录+客户管理+备注系统，让用户尽快能用起来
2. **边用边完善**：用户在使用过程中会提出更贴近实际的需求，避免一次性开发太多
3. **数据库设计先行**：最重要的是客户表+分配表+备注表的设计，需要仔细规划
4. **权限最重要**：多角色权限（顾问/主管/管理员）必须第一阶段完成，否则后续很难补

### 9.4 系统特有交互设计要点

这些是从参考系统深度分析中提炼出的用户体验设计，对实际业务效率影响很大：

#### 9.4.1 快捷备注按钮（18个）
客户详情页的右侧备注区有18个快捷按钮：
```
未接 | 不需要了 | 通话中 | 拒接 | 接了就挂 | 无法接通
用户正忙 | 贷款即挂 | 空号 | 停机 | 关机 | 外地号码
现在不方便接听 | 不是本人申请 | 微信待通过 | 已发短信 | 已加微信
```
**实现方式**：每个按钮对应一个预设文本，点击后自动填入备注框，减少顾问打字量。这是提升录入效率的关键功能。

#### 9.4.2 备注提示词
顾问输入备注时，系统展示引导提示：
> 1、户籍地在哪 2、上班还是做生意 3、工资多少/营业执照多长时间 4、是否有社保和公积金 5、自己和配偶是否有房车和保单 6、负债和征信情况 7、初步判断可以做哪些银行 8、预约什么时候上门

**实现建议**：在备注文本框上方用占位符/placeholder文本展示，引导新人规范备注格式。

#### 9.4.3 贷款资质文本框
客户详情中有一个自由文本区"客户资质"，顾问可填写有利于贷款审批的关键信息摘要。系统提供"修改"和"修改历史"功能。

#### 9.4.4 客户详情展示方式
参考系统使用**右侧iframe展开**而非弹出窗口，好处：
- 不遮挡列表，可以同时参考其他客户
- 支持"上一位/下一位"翻页浏览
- 页面状态不丢失

**建议新系统采用**：右侧抽屉（Drawer）组件，保留参考系统的交互习惯。

#### 9.4.5 状态变更触发记录
客户状态变更时：
- 备注文本框弹出"状态已更新为XXX"自动提示
- 顾问需要在备注中说明状态变更原因
- 状态变更自动记录在分配记录中

---

**绝对不能省略的设计**：
- 备注只能增加，不能修改/删除（历史可信度）
- 客户状态变更要有变更日志
- 所有敏感操作要有操作记录（谁/何时/从哪个IP/做了什么）
- 手机号在数据库中建议加密或哈希存储
- 公共池分配必须有并发锁（防止同一客户被两个顾问同时抢）

**不建议做的**：
- 一开始就上微服务（20人用，单体架构足够）
- 一开始就买昂贵的专有服务器（云服务器弹性更好）
- 一开始就做移动端（PC端先跑稳）
- 自研通话录音系统（成本极高，对接第三方更合算）

### 9.2 开发顺序建议

```
第1步：数据库设计（核心表结构）
第2步：用户认证（登录/权限/组织架构）
第3步：客户基础管理（增删改查）
第4步：客户列表+搜索+筛选
第5步：备注系统
第6步：公共池+分配机制
第7步：在审件管理
第8步：数据统计报表
第9步：外部接口集成（通话/短信）
第10步：高级功能（AI/视频等）
```

## 十二、公共池抓取规则设计

### 12.1 抓取方式

- 2-5星同理，各自独立配置天数阈值

- 待跟进状态：超过N天未跟进 → 自动抓入公共池
- 待签约状态：超过N天 → 自动抓入
- 已放款状态：超过N天 → 自动抓入
